# Philosophy

SolarUI est construit sur quelques principes intentionnels. Pas des conventions arbitraires — chacun répond à un vrai problème visuel ou d'interaction. Cette page les documente pour que le système reste cohérent au fil du temps.

---

## Built on shadcn/ui — open code, no lock-in

SolarUI repose sur shadcn/ui, et ce choix est philosophique autant que technique. shadcn/ui n'est pas une librairie de composants au sens classique du terme : c'est une façon de construire ta propre librairie. Plutôt que d'installer un package opaque et de te battre contre ses API, tu reçois le code source directement dans ton projet. Tu le possèdes. Tu peux le lire, le modifier, l'adapter — sans workarounds, sans wrapper, sans lock-in.

> "This is not a component library. It is how you build your component library."

C'est exactement ce que SolarUI fait : il s'appuie sur cette fondation ouverte pour construire un système de design cohérent, que tu contrôles entièrement.

## Controls share a single height

Buttons and inputs must always have the same height. When a button sits next to an input, they form a single visual unit. If their heights differ, that unit breaks — the eye reads two separate elements instead of one composed control.

<DoAndDont>
  <Do label="Do — same height">
    <div className="flex w-full max-w-xs gap-2">
      <Input placeholder="Email address" />
      <Button>Subscribe</Button>
    </div>
  </Do>
  <Dont label="Don't — mismatched heights">
    <div className="flex w-full max-w-xs items-center gap-2">
      <input
        placeholder="Email address"
        className="flex h-10 w-full rounded-md border border-(--color-input) bg-(--color-background) px-3 text-sm"
      />
      <button className="inline-flex h-6 shrink-0 items-center rounded-lg bg-default-12 px-2.5 text-xs font-medium text-brand-1">
        Subscribe
      </button>
    </div>
  </Dont>
</DoAndDont>

This is enforced at the token level. Both `Button` (default size) and `Input` use the same `h-control` class, which maps to the `--height-control` CSS variable. Changing one automatically changes both.

```css
/* globals.css */
--height-control: 2.25rem; /* 36px */
```

```tsx
// button.tsx  →  size default
"h-control gap-1.5 px-2.5 ..."

// input.tsx
"flex h-control w-full ..."
```

This also applies across sizes: a small button (`size="sm"`) paired with a small input should share the same height token for that size tier.

## Regrouper, catégoriser, ordonner

Chez Solar, on accorde une attention particulière à la façon dont les composants et les icônes sont triés et regroupés. Ce n'est pas une question d'esthétique — c'est une décision structurelle qui a des effets concrets sur la qualité du système.

### Loi de proximité

Les éléments proches sont perçus comme liés. En regroupant des composants qui partagent un rôle, un contexte ou un comportement, on crée une grammaire visuelle implicite : l'utilisateur comprend les relations entre les pièces sans qu'on ait besoin de les lui expliquer.

Un bouton `ghost` isolé dit peu de chose. Placé dans une catégorie "Actions secondaires" aux côtés de `link` et `subtle`, il prend un sens immédiat. La proximité porte la signification.

### Gouvernance et périmètre

Regrouper, c'est aussi délimiter. Une catégorie bien définie dit non seulement *ce qu'elle contient*, mais aussi *ce qu'elle n'accepte pas*. Cela évite le glissement : un composant mal classé finit par être mal utilisé, dupliqué, ou oublié.

Les catégories sont des contrats implicites. Elles rendent le système gouvernable à mesure qu'il grandit.

### Scalabilité cognitive

Un système de design grossit vite. Sans structure, la charge mentale explose : retrouver un composant, savoir où en ajouter un nouveau, comprendre ce qui existe déjà — tout devient coûteux. Des catégories claires réduisent ce coût. Elles permettent à quelqu'un qui découvre le système de se repérer en quelques secondes, sans avoir tout en tête.

### Auditabilité

Quand les composants sont bien rangés, les trous se voient. Une catégorie "Feedback" qui ne contient qu'un `Toast` signale immédiatement qu'il manque peut-être une `Alert`, un `Banner`, un état `Empty`. La structure révèle les lacunes — là où un inventaire plat les dissimule.

### Cohérence sémantique

Un bon regroupement force une question utile : *ces éléments partagent-ils vraiment quelque chose ?* Si la réponse est non, c'est souvent le signe que le composant lui-même est mal conçu, ou qu'il remplit plusieurs rôles qu'il faudrait séparer. Catégoriser oblige à clarifier l'intention.
